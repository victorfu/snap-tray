# JP / KR / TH Website Localization Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add Japanese (`ja`), Korean (`ko`), and Thai (`th`) to the snaptray.cc marketing site and user docs, reusing the existing en/zh-tw i18n mechanism.

**Architecture:** First refactor the hardcoded 2-language switcher + hreflang into a data-driven loop over a new `_data/languages.yml` registry (route-driven, so untranslated pages never show dead links). Then add each language as a self-contained bundle: i18n data file → routes + marketing page wrappers → nav labels + user-doc prose pages. Developer docs stay English.

**Tech Stack:** Jekyll (kramdown, Liquid), YAML data files, vanilla JS (`releases.js`), Ruby (build + verification scripts).

**Spec:** `docs/superpowers/specs/2026-06-13-i18n-ja-ko-th-design.md`

Platform caveat carried into every language: Recording + OCR + GIF/WebP = macOS/Windows only; Linux = Ubuntu 22.04 X11 beta (capture/annotation only); Wayland unsupported.

- [ ] **Step 3: Refactor the language switcher to loop the registry**

Replace the entire contents of `docs/_includes/language-switcher.html` with:

```liquid
{% assign lang = page.lang | default: site.default_lang %}
{% assign route = site.data.routes[page.route_key] %}
{% assign raw_url = page.url | default: '/' %}

<div class="language-switcher" aria-label="{{ site.data.i18n[lang].language.label }}">
{% for l in site.data.languages %}
  {% assign code = l.code %}
  {% assign target = '' %}
  {% if route and route[code] %}
    {% assign target = route[code] %}
  {% elsif code == 'en' %}
    {% assign target = raw_url | replace_first: '/zh-tw', '' %}
    {% if target == '' %}{% assign target = '/' %}{% endif %}
  {% elsif code == 'zh-tw' %}
    {% if raw_url == '/' %}
      {% assign target = '/zh-tw/' %}
    {% elsif raw_url contains '/zh-tw/' %}
      {% assign target = raw_url %}
    {% else %}
      {% assign target = '/zh-tw' | append: raw_url %}
    {% endif %}
  {% endif %}
  {% if target != '' %}
  <a href="{{ target | relative_url }}" class="lang-link {% if lang == code %}is-active{% endif %}">{{ l.name }}</a>
  {% endif %}
{% endfor %}
</div>
```

Rationale: en/zh-tw keep their existing URL-heuristic fallback for pages with no `route_key`; ja/ko/th render **only** when the page's route has an entry for them, so English-only developer pages show just EN/繁中.

- [ ] **Step 4: Refactor the hreflang block to loop the registry**

In `docs/_includes/head.html`, replace lines 36-40 (the `{% if route %}` hreflang block) with:

```liquid
{% if route %}
{% for l in site.data.languages %}{% if route[l.code] %}
<link rel="alternate" hreflang="{{ l.hreflang }}" href="{{ route[l.code] | absolute_url }}">
{% endif %}{% endfor %}
<link rel="alternate" hreflang="x-default" href="{{ route.en | absolute_url }}">
{% endif %}
```

- [ ] **Step 2: Extend `resolveLang()`**

Replace the `resolveLang()` function body so the new codes resolve from `<body data-lang>` / `<html lang>`:

```javascript
function resolveLang() {
  const source = (document.body?.dataset.lang || document.documentElement.lang || 'en').toLowerCase();
  if (source.startsWith('zh')) {
    return 'zh-tw';
  }
  if (source.startsWith('ja')) {
    return 'ja';
  }
  if (source.startsWith('ko')) {
    return 'ko';
  }
  if (source.startsWith('th')) {
    return 'th';
  }
  return 'en';
}
```
